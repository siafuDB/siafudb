The `Mutation` is the atomic unit that travels on the wire. This document defines the normative wire format. See [Mutations](/concepts/mutations) for the conceptual overview.

## Status

**Draft.** Fields listed here are the expected shape pulled from [02 — Graph Sync Protocol](/research/02-graph-sync-protocol). Breaking changes still possible before 0.1.0.

## Fields

| Field | Type | Required | Notes |
|-------|------|----------|-------|
| `id` | UUIDv7 | yes | Time-ordered for natural sort |
| `source` | string | yes | Identity URI of the author |
| `op` | enum | yes | `vertex-create` \| `vertex-update` \| `vertex-delete` \| `edge-create` \| `edge-update` \| `edge-delete` |
| `target` | GraphRef | yes | Vertex or edge being changed |
| `payload` | CBOR | yes | New state or diff |
| `clock` | VectorClock | yes | Causal metadata |
| `signature` | bytes | optional | Ed25519 over the canonical CBOR encoding |

## Encoding

Canonical CBOR (RFC 8949 deterministic encoding). Signatures cover the bytes of the canonical form, so any re-serialisation preserves the signature.

## Batches

A `MutationBatch` wraps an ordered list of mutations plus a batch-level clock. Batches apply all-or-nothing. Within a batch, ordering is preserved; across batches, per-mutation clocks determine ordering.

## Open questions

- Compression of payloads for large binary vertices (consider zstd framing).
- Signature aggregation for batches with a single author.
- Schema versioning inside the payload — embed a schema hash, or trust the receiver's schema registry?

These are tracked in the [Decision Log](/research/03-decision-log).
